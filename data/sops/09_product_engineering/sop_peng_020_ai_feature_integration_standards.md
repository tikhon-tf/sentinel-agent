# AI Feature Integration Standards

## 1. Purpose and Scope

This Standard Operating Procedure (SOP) establishes the binding architectural, procedural, and governance standards for integrating any artificial intelligence (AI) or machine learning (ML) model into the Meridian Health Technologies product ecosystem. As an entity operating at the intersection of clinical decision support, financial services automation, and analytics, the reliability, transparency, and auditability of our AI feature integrations are foundational to patient safety, financial integrity, and regulatory compliance.

This document applies to all AI/ML features developed internally, procured from third-party vendors, or consumed via API-based model-as-a-service platforms that are intended for deployment within any Meridian Business Unit, including Clinical AI (FDA-cleared diagnostic imaging), MedInsight Analytics (population health and care gap identification), and the AI-driven components of the Financial Services platform. It governs the entire lifecycle of an AI integration: from initial design and architectural review through model serving, fallback behavior definition, user transparency implementation, and ongoing monitoring.

### 1.2 Scope

The standards defined herein apply to:

- **All Personnel:** Product Managers, Software Engineers, Machine Learning Engineers, MLOps Specialists, Site Reliability Engineers (SREs), Quality Assurance (QA) Analysts, and User Experience (UX) Designers within the Product & Engineering business unit.
- **All Integration Patterns:** Synchronous API calls, asynchronous inference via message queues, edge deployment of compiled models, and batch scoring pipelines.
- **All Environments:** Local development (`dev`), shared integration (`int`), staging (`stg`), and production (`prd`).
- **All Data Types:** Structured clinical data (e.g., lab results, vitals), unstructured clinical data (e.g., physician notes, radiology images), patient-generated health data, and financial transaction data that is linked or linkable to an individual.
- **All Model Types:** Deterministic statistical models, classical ML models, deep neural networks, large language models (LLMs), and ensemble systems.

This SOP is subordinate to Meridian’s overarching Data Governance Policy (`SOP-DGOV-001`), Information Security Policy (`SOP-ISEC-005`), and Model Risk Management Framework (`SOP-MRMF-101`).

## 2. Definitions and Acronyms

The following terms are critical to the understanding and implementation of this SOP.

## 4. Policy Statements

The following high-level policies are non-negotiable mandates for any AI feature integration.

**POL-020.1: Data Minimization and De-identification by Default.** Any dataset used for non-production purposes (e.g., development, testing, training) must undergo rigorous de-identification per HIPAA Safe Harbor or Expert Determination before being made available. Production data containing PHI may only be processed by an AI feature if there is a direct, documented, and CAG-approved clinical or operational necessity.

**POL-020.2: Mandatory Architecture Review.** No AI feature may be deployed to production without a pre-approved Architecture Decision Record (ADR) documenting the integration pattern, model serving topology, data lineage, and fallback strategy. The ADR must be reviewed and signed off by the Technical Owner and, if applicable, the Clinical Governance Lead.

**POL-020.3: Transparency and Explainability.** Any user interface displaying output derived from an AI/ML model must (a) clearly indicate its synthetic origin, and (b) provide a just-in-time link to a Model Factsheet document (adhering to `SOP-MRMF-110`) describing the model’s intended use, performance characteristics, and limitations.

**POL-020.4: Graceful Degradation and Fail-Safe.** All AI integrations must implement a deterministic fallback behavior. Under no circumstances shall a failure in the AI prediction subsystem result in an unhandled exception, an empty state masquerading as a valid result, or a blocking action in a real-time clinical or financial workflow.

**POL-020.5: Continuous Monitoring.** Every AI integration must emit standardized telemetry to Meridian’s central observability platform (Datadog). This telemetry must include model prediction metrics (confidence, latency) and operational metrics (error rate, drift metrics) as formal SOC 2 control evidence.

## 5. Detailed Procedures

This section outlines the mandatory step-by-step operational procedures for integrating an AI feature. Each integration must be broken into four phases: Design, Implementation, Verification, and Deployment.

### 5.1 Phase 1: Design and Risk Assessment

A cross-functional design review, known as the AI Integration Readiness Review (AIRR), must be convened for every new AI feature or major change to an existing one.

#### 5.1.1 Integration Pattern Selection and ADR

The ML Engineer, in conjunction with the Application Software Engineer and an MLOps/SRE representative, must select an integration pattern from the Meridian-approved catalog. The decision must be documented in an ADR, version-controlled in the service’s repository.

**Approved Integration Patterns:**

1.  **`sync-rest` (Synchronous REST):** The application server makes a synchronous POST request to a SageMaker real-time inference endpoint and blocks the user request thread until a response or timeout.
    - *Applies to:* Real-time, low-latency predictions where the result is essential before the user can proceed (e.g., suspected pathology detection on a single uploaded image). This pattern demands the strictest fallback behavior.
2.  **`async-mq` (Asynchronous via Message Queue):** The application server publishes an inference task to a RabbitMQ queue. A dedicated inference worker consumes the task, executes the model, and writes the result to a database, which then triggers a notification to the user.
    - *Applies to:* Longer-running, complex inferences (e.g., batch analysis of a patient’s multi-year history, multi-modal fusion models) where an immediate result is not needed. The user transparency component must manage user expectation regarding processing time.
3.  **`edge-deploy` (Edge/Client-Side Inference):** A compiled model (e.g., ONNX Runtime, CoreML) is bundled with the client application and runs entirely on the user’s device.
    - *Applies to:* Ultra-low-latency needs for non-sensitive pre-processing (e.g., image quality detection “Good View” before upload). PHI cannot be persisted on the edge. This pattern has a lower fallback risk as the model’s failure doesn't block the core application workflow.
4.  **`hail-mary` (Human-in-the-Loop):** An extension of the `async-mq` pattern. When the model’s confidence score falls below a pre-defined threshold, the inference result and source data are not automatically surfaced to the end-user. Instead, they are routed to a Meridian-internal review queue (Meridian Clinical Review Workbench) for adjudication by a qualified clinical auditor.
    - *Applies to:* High-risk, low-certainty clinical use cases as mandated by the CAG for EU MDR Class IIa and above devices.

#### 5.1.2 Fallback Behavior Specification

For each selected integration pattern, a fallback behavior matrix must be defined and approved in the AIRR. The UX Designer must prototype the fallback user experience.

**Mandatory Fallback Behavior Matrix Elements:**

| Failure Mode | Trigger Condition | System Response | User Interface (UI) |
| :--- | :--- | :--- | :--- |
| **Model Timeout** | Inference service response time > SLA (e.g., 2000ms for `sync-rest`) | Terminate connection. Log `ERROR` event. | Clear error banner: “Analysis could not be completed in time. Please try again.” No partial or ambiguous data shown. |
| **Model Error (5xx)** | Inference service returns HTTP 5xx or equivalent gRPC error | Retry 1 time. If still failure, log `CRITICAL` event, trigger P3 incident alert. | Same as timeout, but with a unique `Error ID` for support correlation. |
| **Low Confidence** | Model prediction confidence < pre-defined CAG-approved threshold (e.g., <85%) | For `sync-rest`: Return prediction with confidence score. For `hail-mary`: Route to review queue; do not return. | For `sync-rest`: Display output with a visual “Low Confidence” qualifier and a recommendation for clinical review. For `hail-mary`: Inform the user the analysis is “under review” and will be available within a 4-hour SLA. |
| **Pre-Processing Error** | Input data fails schema validation (e.g., wrong file format, missing required fields, data type mismatch) | Block pipeline. Log `WARN` event with structured error details. | Specific, actionable, non-technical error message guiding the user to correct the input (e.g., “The uploaded file is not a supported DICOM image. Please select a valid .dcm file and try again.”) |
| **Data Drift Alert** | A real-time data drift metric (e.g., KL Divergence) exceeds a threshold defined during model validation | Log `WARN` event and inject a non-blocking monitoring annotation into the response payload. | No immediate user-facing impact. This is an operational back-end signal. |

### 5.2 Phase 2: Implementation and Instrumentation

The implementation phase must adhere to the ADR and the Meridian Secure Development Lifecycle (`SOP-SEC-003`).

#### 5.2.1 Model Serving Integration

All models must be packaged in a Meridian-standard Docker container that inherits from the approved `meridian-ml-base` image. This base image enforces standard AWS SageMaker MME (Multi-Model Endpoint) or Single-Model Endpoint serving protocols.

**Container Mandates:**
- The `Dockerfile` for the model server must implement the `/invocations` (for `sync-rest`) and `/ping` (for health checks) endpoints per the SageMaker hosting specification.
- The container must run as a non-root user.
- All confidential parameters (API keys, database connection strings) must be injected via AWS Secrets Manager or HashiCorp Vault and never hardcoded or stored in environment variables directly.

A model `serving.properties` file must be included in the model artifact tarball, defining:
- `inference.address=http://0.0.0.0:8080/invocations`
- `management.address=http://0.0.0.0:8080/ping`
- `model_name`, `model_version`, `framework` (e.g., PyTorch 2.0)

#### 5.2.2 De-identification Pipeline Standard

Any pipeline that moves data from a production PHI zone to a lower environment (e.g., `int`, `dev`) for model training or testing must pass through the **Meridian De-Id Service (MDIS)**.

The MDIS API call must be explicit and logged:
`POST https://mdis.internal.meridian.com/v1/deidentify`

The request payload includes the source data and a `method` parameter, which must be one of:
- `safe_harbor_strict`: Removes all 18 identifiers with exact-match logic. Use for training data.
- `safe_harbor_fuzzy`: Removes identifiers, truncates zip codes to 3 digits, and obfuscates dates by shifting them by a random offset per patient, ensuring chronological order is preserved. Use for algorithmic development where temporal relationships matter.
- `expert_determination`: Invokes a specialized statistical module approved by the Chief Data Officer. This method returns a unique `deid_run_id` and a formal risk assessment report. Use only for specific, approved secondary research.

#### 5.2.3 User Transparency Component (UI/UX)

The lead Software Engineer must integrate the approved `<ai-transparency-banner>` UI component from the Meridian Design System (`@meridian/ui-components` v3.1+).

**Technical Implementation Details:**
- The component must fetch the model details from the `model_factsheet_url` prop, which is obtained from the inference response header: `x-meridian-model-factsheet-id`.
- For `hail-mary` integrations, the component dynamically renders a countdown timer to the 4-hour review SLA, polling a status endpoint every 60 seconds. Polling must use exponential backoff (retry at 1s, 10s, 60s, 10m) on failure to avoid overwhelming the backend.
- The component’s `aria-label` must be set to `"AI-generated content explanation"` for accessibility compliance (`SOP-UX-002`).

### 5.3 Phase 3: Verification and Validation

No AI integration proceeds to production without passing the QA Analyst’s “Go/No-Go” checklist.

#### 5.3.1 Integration Testing

The QA team must execute a structured test plan, `SOP-PENG-020-TP-1`, in a staging environment. This plan must include:

1.  **Golden Path Test:** A valid input returns a successful 200 OK response with a correctly formatted prediction payload within the SLA.
2.  **Authentication Failure Test:** Request with an invalid IAM role or expired token receives an HTTP 403, and this is correctly handled by the main application with a “session expired” message.
3.  **Malformed Input Test:** Feeding a corrupted DICOM file or a JSON payload missing `patient_id` returns an HTTP 400 with a descriptive error from the pre-processing layer, not a 500 error from the model server.
4.  **Fallback Matrix Test:** A specific, isolated system test must be run for **each** failure mode defined in section 5.1.2. This is achieved by mocking the downstream inference service with tools like WireMock or by deploying a “chaos” mode of the model container that can simulate timeouts, errors, and low confidence on command.
5.  **Transparency Audit:** The UX audit must verify that the AI disclosure is prominent, non-dismissable for the duration of the result display, and links to a valid, populated Model Factsheet.

### 5.4 Phase 4: Progressive Deployment

All deployments must follow a Progressive Delivery model orchestrated by Argo Rollouts. A standard deployment uses a Canary strategy.

#### 5.4.1 Canary Rollout Steps for AI Inference Services

1.  **T=0m:** Deploy new `v2.0` model serving container. Argo Rollouts deploys the new `Rollout` object. Weight is set to 0% for `v2.0`, 100% for `v1.9`.
2.  **T=5m:** After passing liveness and readiness probes, the rollout controller automatically shifts 5% of production traffic to `v2.0`.
3.  **T=20m:** The MLOps Dashboard, monitored by the SRE on-call, must validate that the `v2.0` error rate is < 0.01%, and the p95 prediction latency is not more than 10% greater than `v1.9`’s baseline. If these metrics violate thresholds, the rollout controller is configured to automatically initiate a full rollback to `v1.9` within 60 seconds.
4.  **T=25m:** If the 20-minute `v2.0` analysis is clean, the SRE on-call manually advances the canary to 25% via a confirmed action in the Argo Rollouts dashboard.
5.  **T=2h:** After 2 hours of clean telemetry at 25% traffic, the rollout automatically progresses to 100% and stabilizes. The old `v1.9` replicas are scaled down and terminated after a 10-minute grace period.

### 6.2 NIST AI RMF MEASURE Function: Technical Safeguards

This section details the mandatory observable controls that provide quantitative evidence of system safety and trustworthiness.

| Control ID | Control Description | RMF Measure Subcategory | Implementation (Meridian Tool: Datadog + AWS Config) |
| :--- | :--- | :--- | :--- |
| **AI-TS-01** | **Inference Latency SLI/SLO** | MEASURE 2.6 (Performance) | An AWS CloudWatch High-Resolution Metric for `ModelInvocationLatency` must be defined per model. The SLO for `sync-rest` patterns is: 99.9% of requests must be served in < 2000ms over a rolling 28-day window. A Datadog SLO widget tracks this. A breach triggers a P2 alert. |
| **AI-TS-02** | **Input Data Drift Monitoring** | MEASURE 2.9 (Data Quality) | An automated SageMaker Model Monitor job runs daily for every production `sync-rest` and `async-mq` endpoint. It calculates baseline drift metrics (KL Divergence, KS Statistic) from the training dataset. A `WARN` threshold is set at 0.1, and a `CRITICAL` threshold at 0.3. Breaches trigger auto-generated Jira tickets in the `ML-OPS` project. |
| **AI-TS-03** | **Model Explainability Score** | MEASURE 2.8 (Explainability) | Shapley Additive Explanations (SHAP) values are calculated and logged alongside each inference for high-risk clinical models. The mean SHAP for the top 3 features is logged as a distribution metric. Unexplained, anomalous attribution deviations are a `WARN` alert. |
| **AI-TS-04** | **Confidence Score Distribution** | MEASURE 2.9 (Uncertainty) | A histogram metric `ModelConfidenceDistribution` must be emitted per inference. This is visualized on a dedicated model dashboard. Drift in the 99th percentile of low-confidence predictions triggers a model re-evaluation task. |

## 7. Monitoring, Metrics, and Reporting

Continuous oversight is a fundamental, auditable requirement. This section defines the “what,” “how,” and “who” of our monitoring strategy.

### 7.1 Golden Signals Dashboard

The primary operational dashboard for each AI Feature is its “Golden Signals” dashboard in Datadog. It must contain the following four rows, with the specified SLO widgets.

1.  **Traffic:** Requests per second (RPS), broken down by `model_version` and `response_code`.
    - *Widget:* `Request Rate` (timeseries).
2.  **Latency:** p50, p95, and p99 inference latency (milliseconds).
    - *Widget:* `Service Level Objective (SLO)` for Latency SLI as defined in AI-TS-01.
3.  **Errors:** Error rate (percentage of 5xx responses and application-level prediction errors).
    - *Widget:* `Error Rate` (bars) with a threshold-based `Monitor` for P1/P2 alerting.
4.  **Saturation:** Model server CPU/Memory utilization per container, and RabbitMQ queue depth (for `async-mq`).
    - *Widget:* `Resource Utilization` (heat map).

## 8. Exception Handling and Escalation

Adherence to this SOP is expected. However, a formal exception process exists for cases where strict compliance may hinder critical business or clinical innovation.

### 8.1 Exception Request Process

Any team member may raise an Exception Request (ER). The process is as follows:

1.  **Initiation:** Requestor completes the `SOP-PENG-020-ExceptionRequest_v2` form in Jira Service Management. The form requires a detailed justification, a description of the compensating control, and a proposed timeframe for the temporary exception.
2.  **Technical Impact Assessment:** The Director of MLOps (or delegate, as Technical Owner of this SOP) reviews the request and creates a “Risk & Impact Assessment” within the Jira ticket. This is a mandatory sub-task.
3.  **Governance Review:**
    - If the exception pertains to de-identification (POL-020.1), the Chief Data Officer must approve.
    - If the exception pertains to fallback behavior or clinical transparency (POL-020.3, POL-020.4), the Clinical Advisory Group (CAG) Lead must approve.
    - For all other exceptions, or if the compensating control involves accepting a specific, measurable re-identification risk, the VP of Engineering (Executive Owner) is the final approver.
4.  **Voting and Resolution:** The approving body votes within the Jira ticket. A simple majority is required for approval. All approvals, including non-approvals, must be accompanied by a written rationale.
5.  **Sunsetting:** Every approved exception has an immutable Expiration Date, enforced by a Jira Automation workflow. 30 days before expiration, the automation pings the requestor and the VP of Engineering to either resolve the condition, request a one-time extension (which requires a new full approval cycle), or plan for the feature’s rollback. Expired, unresolved exceptions are automatically escalated to the CEO, Dr. Sarah Chen.
